1. Executive Summary

ItemSmart is an SKU-level sales planning tool within the Impact Analytics (IA) Smart Platform. It uses forecast data (12+ month horizon), planning metrics (e.g., WOS, ST%), and receipt recommendations to support buy placement and item planning. Users view past, present, and future demand alongside constraints and SKU attributes over a configurable time horizon.

This PRD consolidates product requirements derived from ItemSmart-related user stories in the MTP Jira project and supporting Confluence documentation (Data Flow, Architecture, Integration). It is intended for product, engineering, and QA to align on scope, acceptance criteria, and priorities.

2. Product Overview

2.1 Purpose

  • Primary: Enable planners to plan at item/SKU–week level with forecast, KPIs, and receipt recommendations in one place.
  • Outcomes: Streamlined workflow, data consistency across planning levels, and reliable handoff to Ordering/OMS, AssortSmart, and PlanSmart.

2.2 Planning Context

Aspect Description
Planning type Bottom-up, pre-season (12–18 months prior)
Planning level Item–week level (SKU × channel × time)
Key inputs Forecast (IAF), LY/LLY, actuals, item facts, constraints
Key outputs Receipt recommendations, plan values (WP), handoff to downstream systems

2.3 User Personas

  • Planner — Day-to-day editing of KPIs, viewing dashboards, applying filters, managing product groups.
  • Executive / DPM — Review and approval; high-level visibility via KPIs and summaries.
  • Admin / Superuser — Configuration, access control, and integration settings.

2.4 Ecosystem Position

ItemSmart sits within the MTP ecosystem with the following integration points (from Confluence: ItemSmart Integration):

  • Upstream: PlanSmart (assortment plan, PO-level inputs); optional PIM (e.g., Arhaus) for item facts.
  • Downstream: Ordering, OMS (InventorySmart), AssortSmart, PlanSmart (depending on client setup).
  • Use cases: ItemSmart-only for core planning; PlanSmart + ItemSmart; PlanSmart + AssortSmart + ItemSmart/OMS (e.g., Carter’s).

3. Goals & Success Criteria

Goal Success Criteria
Usability Planners can open ItemSmart from the IA platform and reach the right planning view with minimal clicks.
Data consistency Edits at any hierarchy level (SKU/class/department; week/month) propagate correctly via defined aggregation/disaggregation rules.
Integration Clear, documented flows and cadence for Ordering ↔ ItemSmart, ItemSmart → AssortSmart, ItemSmart → PlanSmart.
Performance Dashboard and grid load within agreed SLAs; no material impact on login or refresh times.
Accessibility Features meet agreed accessibility standards and work across supported devices.

4.8 Data Flow & Versions (from Confluence)

ItemSmart uses multiple data versions; behavior is defined in Data Flow in ItemSmart. Summary:

Version Purpose Update mechanism
IAF Forecast-driven demand; initial source for WP Weekly/daily refresh; baseline discount filter; no WP edits copied back.
LY Last year (1y + 9m history) Weekly refresh; full weeks only; not editable from tool.
LLY Last last year (2y + 9m history) Weekly refresh; full weeks only; not editable from tool.
Actual Past actuals (e.g., 9m prior to current week) Shown as WP for actualized periods; non-editable; weekly refresh.
WP Working plan (user-editable) Seeded from IAF; only seeding, match-with, and manual edits update WP; weekly refresh does not overwrite WP; actuals replace WP for completed full weeks.

Relevant requirements:

ID Requirement Acceptance Criteria
DATA-1 IAF generation and refresh per defined rules (SKU selection, horizon, baseline discount). As per Data Flow: 8 weeks before launch to max date; 26 weeks past exit or 18 months; weekly/daily refresh as configured.
DATA-2 WP seeded from IAF; receipt recalculated on the fly where specified. Seeding and receipt logic as documented.
DATA-3 Actualized weeks replace WP in UI (and optionally in DB). No WP edits overwrite actuals; actuals overwrite WP for completed weeks.
DATA-4 Date selection follows Gregorian calendar. All date logic calendar-consistent.

4.9 Item Facts & PIM Sync (e.g., Arhaus)

ID Requirement Source Acceptance Criteria
PIM-1 New items: item facts from PIM feed; daily ingestion updates ItemSmart and KPIs. MTP-51042 New SKUs receive PIM facts and refresh correctly.
PIM-2 Existing items, no ItemSmart edit: PIM remains source; daily feed updates item facts and KPIs. MTP-51042 Unedited items stay in sync with PIM.
PIM-3 Existing items, edited in ItemSmart: ItemSmart is source of truth for edited facts; feed updates ignored for those facts. MTP-51042 Edited facts not overwritten by feed.

PIM-5 Conflict handling when ItemSmart edits and PIM differ. MTP-51042 Clear, documented behavior (e.g., ItemSmart wins for edited fields).
PIM-6 (Future) User choice of source of truth after PIM is updated from deviation report. MTP-51043 Documented and implemented when prioritized.

6. Architecture & Technical Context

  • Front-end: React; NGINX for load balancing and static assets.
  • Backend: FastAPI (Python); Core Service (auth, orchestration) + ItemSmart Backend (item/inventory operations).
  • Database: PostgreSQL (primary + read replica for reporting); BigQuery for analytics/warehouse.
  • Data sync: Ordering/OMS ↔ ItemSmart designed via CDC/outbox and event-driven sync (see Confluence TDDs).
  • Application ID: ItemSmart uses APPLICATION_ID and app-level DB segregation per MTP (e.g., item).

Detailed diagrams and deployment notes are in Confluence (e.g., Itemsmart Architecture).
